# HZ11 NO-GO Ledger

Status: initialized with inherited decisions from HZ8/HZ10 speed work.

Do not retry these ideas in the HZ11 default hot path without new evidence.

| Box / Idea | Decision | Reason |
|---|---|---|
| Preserve HZ10 owner-return remote free in HZ11 speed mode | NO-GO for default hot path | HZ11 exists to test current-thread/current-CPU cache reuse. Owner-return is a HZ8/HZ10 correctness/RSS design, not the tcmalloc-speed shape. |
| Route every local free through pagemap metadata | NO-GO for default hot path | HZ10 attribution showed route and ownership validation are fixed costs. HZ11 must use token/cache hits and route only on fallback. |
| Put fail-closed diagnostics on every speed-mode op | NO-GO for default hot path | Checked diagnostics are valuable, but speed mode must not pay marker/state/generation/reason-counter costs by default. |
| Optimize HZ10 into HZ11 in-place | NO-GO | HZ10 is the route/ownership/reclaim research candidate. HZ11 needs a different semantic split and should live in a separate folder. |
| HZ11ThreadCacheFastPath-L0: system-backed token-table front-end as the speed path | NO-GO for the tcmalloc-tier gate (measured) | Implemented + measured. Front-end works (cache hit 100%, token hit 100%, beats glibc/mimalloc/hz10), but RUNS=3 median fixed16/64/256/4096 land at 0.71-0.82 of tcmalloc (gate 0.95). The gap is the free-side token hash: tcmalloc classifies a free by a direct-indexed page table (~1-2 cycles), hz11 must hash because system-malloc objects have arbitrary addresses (no contiguous region to index). The hash is structural to the system-backed L0. Next box: span/pageheap backing -> direct-indexed classification -> close the gap. |
| HZ11SpanBackedClassify-L1: arena/span direct-index classify | NO-GO for the tcmalloc-tier gate (measured) | Implemented + measured against the L0 token lane. The span lane is correct after fixing arena range checks to use integer address arithmetic and adding an overflow returned-list sink, but RUNS=5 fixed64 reaches only 0.739 of tcmalloc (gate 0.95) and improves over the token lane by only ~2-3%. The L0 token-hash diagnosis was too strong: classification is not the main lever. Next box should attribute the whole HZ11 front-end instruction/call/TLS/size-class cost against tcmalloc on the same fixed rows. See docs/no_go/HZ11_SPAN_BACKED_CLASSIFY_L1.md. |
| HZ11FrontEndAttribution-L0 | Attribution GO (conclusion only -- not an optimization GO) | Measured fixed64 instr/op via perf stat -r 3: tcmalloc 87.3, hz11-token 129.4 (+42), hz11-span 135.4 (+48). Gap is instruction COUNT, not efficiency (HZ11 IPC 4.39 > tc 3.86; cycles gap only +6.9). Objdump: wrapper is 3-instr tail-call (H1 RULED OUT). Step-list ranking: H4 (hot-path counter increments, 4-5 RMW/op, ~12-15 instr/op) is the LARGEST single chunk; H5 (pop/push, ~10-12) is 2nd; H2 (TLS, ~6-8) moderate; H3 (size-class, ~4-5) minor. Next box candidate: HZ11StatsCompileGate-L1 (compile-time counter opt-out), but NOT GO'd here. See docs/HZ11_FRONTEND_ATTRIBUTION_L0.md. |
| HZ11StatsCompileGate-L1 | GO as cleanup, NO-GO as primary tcmalloc-gap lever | `HZ11_ENABLE_HOT_COUNTERS=0` is now the default speed lane and diagnostic siblings keep counters on. Fixed64 improved only about 4 instr/op (`hz11-token` 129.4 -> 125.4, `hz11-span` 135.4 -> 131.4) and ops/s moved +1-3%; the target 115 instr/op was not reached. Counter updates were real but much smaller than the attribution estimate because LTO lowered them to compact increments. Next box: HZ11CacheShape-L1; TLSFastPath remains second. See docs/HZ11_STATS_COMPILE_GATE_L1.md. |
| HZ11CacheShape-L1: pointer-top class cache | NO-GO | Added `HZ11_CACHE_TOPPTR` siblings (`libhz11_top.so`, `libhz11_span_top.so`) that keep cached-byte/cap behavior while replacing count-indexed pop/push with a moving top pointer. Smoke passed, but fixed64 instruction count moved the wrong way: token 125.5 -> token-top 138.5, span 131.5 -> span-top 143.5; objdump counts also increased. Token-top showed small throughput noise, but the instruction-count gate failed and complexity increased without a >=5 instr/op win. Next box: HZ11TLSFastPath-L1. See docs/no_go/HZ11_CACHE_SHAPE_L1.md. |
| HZ11TLSFastPath-L1 | GO as hot-path cleanup | Added `HZ11_TLS_FASTPATH` siblings (`libhz11_tlsfast.so`, `libhz11_span_tlsfast.so`) that read `hz11_tls` once and move resolver/init cases to noinline slow helpers. Smoke passed. Fixed64 instruction count dropped clearly: token 125.5 -> 114.5 instr/op, span 131.5 -> 116.5 instr/op; throughput also improved (token 136.8M -> 147.3M ops/s, span 145.6M -> 158.4M ops/s in the RUNS=5 full fixed-size run). Still below tcmalloc's ~87 instr/op, so this is not the final speed solution. Next box should target the remaining public-entry body cost after TLSFastPath. See docs/HZ11_TLS_FAST_PATH_L1.md. |
| HZ11CacheByteAccountingGate-L1 | GO as speed-ceiling cleanup, not default policy | Added `HZ11_CACHE_BYTE_ACCOUNTING=0` siblings (`libhz11_nobytes.so`, `libhz11_span_nobytes.so`) on top of TLSFastPath. Smoke passed. Fixed64 RUNS=10/ITERS=20M: token 114.3 -> 101.3 instr/op and 147.8M -> 154.1M ops/s; span 116.3 -> 103.2 instr/op and 158.1M -> 165.0M ops/s. This prices global cached-byte accounting at ~13 instr/op. Not promoted because it removes the global per-thread byte cap from the hot path; keep as a speed ceiling until a low-cost RSS cap replacement exists. See docs/HZ11_CACHE_BYTE_ACCOUNTING_GATE_L1.md. |
| HZ11CacheLayout-L1 | NO-GO for instruction gate; keep as sibling speed probe | Added `HZ11_CACHE_SOA=1` siblings (`libhz11_soa.so`, `libhz11_span_soa.so`) on top of TLSFastPath + no-bytes. Smoke passed. Fixed64 RUNS=5/ITERS=10M: token about 101 -> 98 instr/op; span about 103 -> 100 instr/op. This confirms the class-cache layout costs instructions, but the win (~3 instr/op) missed the >=4 instr/op and <=96 instr/op gates. Token throughput improved, but span throughput was not stable and can regress. Do not promote; next candidate is HZ11SizeTableStaticInit-L1. See docs/no_go/HZ11_CACHE_LAYOUT_L1.md. |
| HZ11SizeTableStaticInit-L1 | NO-GO for removing the lazy-init guard | Removing `hz11_size_table_ready` from the hot path corrupts the heap under LD_PRELOAD: `ld.so` can call malloc before any shim or size-table constructor runs, so the BSS-zeroed table classifies every cached size as class 0. A priority constructor did not run early enough. Kept the exhaustive 1..65536 class-map smoke test and comments documenting that the guard is load-bearing; changing the flag from volatile to non-volatile did not measurably improve throughput. See docs/no_go/HZ11_SIZE_TABLE_STATIC_INIT_L1.md. |
| HZ11StaticConstSizeClassTable-L1 | NO-GO; correct but slower | Replaced the runtime-filled BSS table with a checked-in const `.rodata` table to avoid constructor ordering and remove the ready check. All smoke lanes and the exhaustive class-map check passed, but fixed64 `hz11-token-soa` regressed about 7% (roughly 157-159M -> 145.6M ops/s). LTO had already made the mutable-table guard effectively cheap; `.rodata` changed codegen/addressing enough to lose throughput. Reverted the const table; keep only the NO-GO record. See docs/no_go/HZ11_STATIC_CONST_SIZE_TABLE_L1.md. |
| HZ11TransferCacheCentralSpan-L1 | GO for the opt-in transfer lane | Implemented `libhz11_span_transfer.so` with a batch transfer cache and bounded central object stack. THREADS=16/RUNS=10 `bench_matrix_malloc`: main_local0 is tcmalloc-parity (455.58M vs 451.51M ops/s), main_r50 is 87.66M vs tcmalloc 42.02M, main_r90 is 62.41M vs tcmalloc 23.47M, with lower post RSS than tcmalloc on these rows. See docs/HZ11_TRANSFER_CACHE_CENTRAL_SPAN_L1.md. |
